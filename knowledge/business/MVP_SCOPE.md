# MVP_SCOPE — V1スコープ定義

## 役割
V1（最初のリリース）でビルドするものとビルドしないものを固定する。
スコープ外の機能追加要求に対して「V2以降」と明示するための根拠。

## 管理対象
- V1のイン・アウトスコープ一覧
- スコープ変更の判断基準
- V2候補の管理

## 書いてよい内容
- イン・アウトスコープの定義と理由
- スコープ変更時の影響評価基準

## 書いてはいけない内容
- 実装コード
- 特定ベンダーの設定値
- 価格設計（→ PRICING.md へ）

## 更新ルール
- スコープ変更は必ず「理由」と「削除または追加するスコープへの影響」を明記する
- V1リリース後は V1アウトスコープへの変更禁止。V2として新規計画する

## V1 でやること

| 機能 | 説明 |
|---|---|
| 予約確認 SMS / メール送信 | クリニックから患者への予約確認通知 |
| 患者予約確認 | 患者がリンクから予約内容を確認する |
| キャンセルポリシー表示 | 同意前に全文を表示する |
| 同意取得 | キャンセルポリシーへの明示的同意（同意ボタン） |
| カード登録 | Stripe SetupIntent でカードを事前登録する |
| 同意ログ保存 | 同意日時・表示文面・IPアドレスをDBに保存する |
| クリニック管理画面 | 予約一覧・来院確認・ステータス確認 |
| キャンセル料請求 | キャンセル時にポリシーに従いcaptureする |

---

## V1 でやらないこと

| 機能 | 理由 |
|---|---|
| LINE連携 | 実装コスト高・V1では SMS / メールで十分 |
| 電子カルテ連携 | 各社ベンダー対応が複雑。V1スコープ外 |
| 保険証連携 | 法的・技術的難易度高。V2以降 |
| AI電話（自動応答） | 誤認識リスクあり。V1では人手フロー優先 |
| 多店舗管理（グループ院） | 権限・請求分離が複雑。V2で設計 |
| 高度分析・ダッシュボード | V1は最小限のステータス確認のみ |
| Stripe Connect 複雑運用 | V1はプラットフォーム直接課金のみ |
| アプリ化（iOS / Android） | Web（PWA相当）で十分。V2以降 |

## V2 候補（優先度順）

1. LINE通知チャネル追加
2. 多店舗管理（グループ院・フランチャイズ）
3. リマインダー自動送信（前日・3時間前）
4. Stripe Connect（クリニックへの直接入金）
5. 電子カルテ連携（主要ベンダー1社から）
