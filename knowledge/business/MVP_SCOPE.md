# MVP_SCOPE — V1スコープ定義

## 役割
V1（最初のリリース）でビルドするものとビルドしないものを固定する。
スコープ外の機能追加要求に対して「V2以降」と明示するための根拠。

## 管理対象
- V1のイン・アウトスコープ一覧
- スコープ変更の判断基準
- V2候補の管理

## 書いてよい内容
- イン・アウトスコープの定義と理由
- スコープ変更時の影響評価基準

## 書いてはいけない内容
- 実装コード
- 特定ベンダーの設定値
- 価格設計（→ PRICING.md へ）

## 更新ルール
- スコープ変更は必ず「理由」と「削除または追加するスコープへの影響」を明記する
- V1リリース後は V1アウトスコープへの変更禁止。V2として新規計画する

---

## MVPの性質

**MVP = 決済なしの予約確認プラットフォーム**

Stripe / Apple Pay / Google Pay / カード登録 / Authorization / Capture は V1 スコープ外。
将来形は PAYMENT_ARCHITECTURE.md に記載する。

---

## V1 でやること

### クリニック側

| 機能 | 説明 |
|---|---|
| 新規予約作成 | 患者名・電話番号・メール・予約日時・内容・キャンセルポリシーを入力 |
| 確認URL発行 | 作成後に患者向け確認URLを発行。画面表示 + コピー |
| 予約一覧 | ステータス別に予約を一覧表示 |
| 来院確認 | 確認済予約を「来院済」に変更 |
| 上部サマリー | 本日の予約 / 確認待ち / 確認済 / 来院済 / キャンセル |

### 患者側

| 機能 | 説明 |
|---|---|
| 予約内容確認 | URLを開き予約日時・内容を確認 |
| キャンセルポリシー確認 | 同意前に全文を表示 |
| 同意チェック | 明示的同意（チェックボックス + 確定ボタン） |
| 予約確定 | 同意完了後に予約を確定状態に更新 |
| QR来院チケット発行 | 確定後にQRコード付き来院チケットを表示 |

### 通信手段

| チャネル | MVP対応 | 備考 |
|---|---|---|
| manual | ✅ 対応 | URL表示 + コピーボタン |
| sms | 🔜 将来対応 | MVP は「準備中」として UI に表示 |
| email | 🔜 将来対応 | MVP は「準備中」として UI に表示 |
| line | 🔜 将来対応 | MVP は「準備中」として UI に表示 |

実送信APIはMVPでは作成しない。

## V1 でやらないこと

| 機能 | 理由 |
|---|---|
| Stripe実装 | 将来形。PAYMENT_ARCHITECTURE.md に設計を記載 |
| カード登録（SetupIntent） | 将来形 |
| Authorization（オーソリ） | 将来形 |
| Capture（請求実行） | 将来形 |
| Apple Pay / Google Pay | 将来形 |
| SMS実送信 | 実装コスト。MVPはmanual URL |
| メール実送信 | 実装コスト。MVPはmanual URL |
| LINE連携 | 到達率高いが実装コスト高。V2以降 |
| 認証（ログイン） | MVP はURL直接アクセス |
| 本番DBスキーマ（Supabase RLS等） | V1デモはインメモリ。本番化時に設計 |
| 管理者権限・マルチテナント | V2以降 |
| 電子カルテ連携 | 各社ベンダー対応が複雑。V2スコープ外 |
| 多店舗管理（グループ院） | 権限・請求分離が複雑。V2で設計 |
| アプリ化（iOS / Android） | Web（PWA相当）で十分。V2以降 |

## 予約ステータス（MVP）

| ステータス | 表示ラベル |
|---|---|
| pending | 確認待ち |
| confirmed | 確認済 |
| visited | 来院済 |
| cancelled | キャンセル |

## V2 候補（優先度順）

1. SMS送信（Twilio等）
2. LINE通知チャネル追加
3. 予約保証登録（Stripe SetupIntent + Authorization）
4. メール自動送信
5. 多店舗管理（グループ院・フランチャイズ）
6. リマインダー自動送信（前日・3時間前）
7. Stripe Connect（クリニックへの直接入金）
8. 電子カルテ連携（主要ベンダー1社から）
