# INCIDENTS — 事故記録

## 役割
発生した事故・ヒヤリハットを記録し、再発防止策の根拠とする。
「同じ事故を2回起こさない」ための正本。

## 管理対象
- 発生した事故・インシデント
- 事故の原因・影響・再発条件
- 採用した防止策とその有効性
- ヒヤリハット（未発生だが発生しうるリスク）

## 書いてよい内容
- 事故の経緯（時系列）
- 根本原因（技術・設計・運用のどれか）
- 影響を受けた患者数・金額
- 採用した防止策
- 防止策の有効性の評価（後日追記）

## 書いてはいけない内容
- 個人を特定する患者情報
- 推測だけの原因分析（確認できた事実のみ書く）
- 防止策の実装コード（→ 各 skills/ ファイルへ）

## 更新ルール
- 事故発生から24時間以内に「経緯」と「暫定対応」を記録する
- 根本原因分析完了後に「原因」「防止策」を追記する
- 防止策実装後30日以内に「有効性評価」を追記する

## 事故テンプレート

```markdown
### INC-YYYY-NNN: タイトル

**発生日**: YYYY-MM-DD
**深刻度**: Critical / High / Medium / Low
**影響**: 患者X名、金額Y円

**経緯**
（時系列で事実のみ）

**根本原因**
（設計/実装/運用/外部のどれか）

**影響範囲**
（どのデータ・患者・クリニックに影響）

**暫定対応**
（発生直後にとった措置）

**恒久防止策**
（設計・コード・運用の変更）

**有効性評価**（実装後30日で追記）
```

## 既知リスク（未発生・ヒヤリハット）

### RISK-001: オーソリ失効の未検知
- 条件: PaymentIntent発行から7日以内に来院確認が行われない
- 影響: captureできず、キャンセル料も回収できない
- 対策: 失効6日前アラート + 来院確認催促通知

### RISK-002: 二重 capture
- 条件: Webhookの再送とAPI再試行が重なる
- 影響: 患者への二重請求
- 対策: capture前のDB排他チェック + 冪等性キー

### RISK-003: 同意バージョン不一致
- 条件: ポリシー更新後に旧バージョンで同意済みの患者がキャンセル
- 影響: 「同意したポリシーと請求が異なる」紛争
- 対策: 同意ログにポリシーバージョンを保存し、キャンセル料計算時に同意時バージョンを使用

### RISK-004: Webhook未着による状態乖離
- 条件: Stripe Webhookがタイムアウト・失敗
- 影響: DB状態が古いまま患者への通知が遅延または誤配信
- 対策: Webhook署名検証 + 失敗時リトライ + 手動再同期コマンド
