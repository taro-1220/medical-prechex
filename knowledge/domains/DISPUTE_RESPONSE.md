# DISPUTE_RESPONSE — 紛争対応フロー

## 役割
チャージバック（クレジットカード会社への異議申し立て）発生時の対応フローと証拠準備の方針を定義する。
「証拠が整っていれば勝てる」状態を設計段階で作る。

## 管理対象
- チャージバック発生時の初動フロー
- 提出すべき証拠の種類と取得方法
- 証拠パッケージの構成
- Stripe Dispute API との連携方針
- 勝訴・敗訴後の処理

## 書いてよい内容
- 証拠として有効な記録の種類
- 各証拠の取得先（どのテーブル・どのイベントログ）
- 証拠提出の期限（Stripe 規定に基づく）
- 対応優先度の判断基準

## 書いてはいけない内容
- 実装コード
- 法的アドバイス（弁護士確認事項として明記）
- 個別クリニック・患者の紛争事例（INCIDENTS.md へ）

## 更新ルール
- 紛争で新しい証拠の有効性が判明した場合に更新する
- Stripe の証拠提出フォームが変わった場合に更新する

## 証拠パッケージ構成

紛争発生時に自動または手動で収集すべき証拠:

| 証拠 | 取得先 | 証明する内容 |
|---|---|---|
| キャンセルポリシー表示文面 | `cancellation_policies.display_text` (同意時 version) | 患者が何に同意したか |
| 同意ログ | `consent_logs` | 誰が・いつ・どの端末で同意したか |
| 予約作成イベント | `event_logs` (`appointment.created`) | 予約が患者自身によって作成されたこと |
| オーソリ完了イベント | `event_logs` (`authorization.succeeded`) | 患者のカードでオーソリが通ったこと |
| キャンセル操作イベント | `event_logs` (`cancellation.initiated`) | 誰が・いつキャンセルしたか |
| キャンセル料計算イベント | `event_logs` (`cancellation.fee_calculated`) | 計算根拠 |
| 通知ログ | `notification_logs` | 患者に通知を送ったこと（未到着との主張への反論） |
| IPアドレス・User-Agent | `consent_logs`, `event_logs` | 操作した端末の特定 |

## 初動フロー（チャージバック発生から7日以内）

1. `dispute.opened` Webhook 受信 → `appointment.status = disputed` に遷移
2. クリニックとプラットフォームの両方に通知
3. 証拠パッケージを自動生成（上表の全項目）
4. Stripe Dispute 提出期限を確認（通常7〜21日）
5. クリニックに「来院確認有無」と「追加証拠（記録・メモ等）」を確認
6. 期限3日前までに証拠を提出

## 勝訴後
- `appointment.status` を `completed` または `cancelled_charged` に戻す
- INCIDENTS.md に「勝訴理由」を記録（有効だった証拠を明記）

## 敗訴後
- 返金処理を完了する
- INCIDENTS.md に「敗訴理由」を記録（不足していた証拠を明記）
- 不足証拠を設計に組み込む（EVENT_MODEL.md / CONSENT_UX.md の更新）

## 紛争リスク評価
- リスク高: no-show（患者が「知らない」と主張しやすい）
- リスク中: 有料キャンセル（金額の妥当性が争点）
- リスク低: 正常来院後の請求

## 関連する状態
- EVENT_MODEL.md: 証拠となるイベントの定義
- CANCELLATION_POLICY.md: 同意文面のバージョン管理
- STATE_MACHINE.md: `disputed` ステータス

## 関連する事故パターン
- 同意ログに `policy_version` がなく、患者が「見ていない」と主張した文面を特定できなかった
- IPアドレスを保存していなかったため「本人操作」を証明できなかった
- 証拠提出期限に気づかずデフォルト敗訴
