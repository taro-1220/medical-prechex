# CLINIC_ONBOARDING — クリニックオンボーディング

## 役割
新規クリニックがシステムを使い始めるまでのフローとデータ設定を定義する。
設定不備のまま本番稼働することを防ぐ。

## 管理対象
- オンボーディングステップとチェックリスト
- Stripe Connect 接続フロー
- キャンセルポリシー設定の必須要件
- 初回設定完了の定義

## 書いてよい内容
- ステップごとの必須アクションと完了条件
- 設定項目の説明と設計上の制約
- Stripe Connect の接続フロー概要
- 初回稼働前チェックリスト

## 書いてはいけない内容
- 実装コード・管理画面の操作手順書
- キャンセルポリシーの内容（クリニックが決める）
- 法的契約書の内容

## 更新ルール
- オンボーディングフローを変更した場合に更新する
- 初回稼働後にトラブルが発生した場合はチェックリストに追記する

## オンボーディングステップ

1. **クリニック基本情報登録**
   - クリニック名・住所・診療科目
   - 担当スタッフアカウント作成
   - 完了条件: `clinic.status = active`

2. **Stripe Connect 接続**
   - Stripe アカウント作成 or 既存アカウント接続
   - 本人確認（KYC）完了
   - 完了条件: `stripe_account.charges_enabled = true`
   - **禁止**: KYC未完了のまま予約受付開始

3. **キャンセルポリシー設定**
   - 設定必須項目: 無料キャンセル期限、キャンセル料率/金額
   - 患者表示文面のプレビュー確認
   - 完了条件: `cancellation_policy.status = approved`
   - **禁止**: ポリシー未承認のまま予約受付開始

4. **テスト予約実行**
   - スタッフ自身の携帯番号・カードでテスト予約
   - オーソリ → 来院確認 → capture のフルフロー確認
   - 完了条件: テスト予約が `completed` に到達

5. **本番稼働承認**
   - 上記4ステップ完了後にプラットフォーム側が承認
   - 承認前は `clinic.mode = test`（本物の患者予約を受け付けない）

## 初回稼働前チェックリスト
- [ ] Stripe アカウントの `charges_enabled` が `true`
- [ ] キャンセルポリシーが承認済み
- [ ] テスト予約のフルフロー完了
- [ ] スタッフ全員がダッシュボード操作に習熟
- [ ] Webhook エンドポイントが本番URLで設定済み

## 関連する状態
- IDENTITY_MODEL.md: clinic_id 発行タイミング
- CANCELLATION_POLICY.md: ポリシー設定の詳細

## 関連する事故パターン
- KYC未完了のまま予約受付を開始し、captureが失敗した
- テスト予約をスキップし、本番でオーソリ失敗が初めて発覚した
- ポリシー未設定のまま稼働し、キャンセル料を請求できなかった
