# NOTIFICATIONS — 通知設計

## 役割
患者・クリニックスタッフへの通知のタイミング・チャネル・必須内容を定義する。
通知の過不足による「知らなかった」を防ぐ。

## 管理対象
- 通知トリガーとなるイベント
- 各通知の送信先・チャネル・必須内容
- 通知ログの保存要件（紛争証拠として使用）
- 通知失敗時のリトライと代替手段

## 書いてよい内容
- 通知トリガーイベントと送信条件
- チャネル選択の優先順位（SMS > メール > プッシュ）
- 必須テキスト要素（省略禁止の情報）
- 通知ログのスキーマ要件

## 書いてはいけない内容
- 通知テンプレートの本文（別ファイルで管理）
- 実装コード・メール配信サービスの設定

## 更新ルール
- 新しいイベントを EVENT_MODEL.md に追加した場合、通知の要否を評価して更新する
- 紛争で「通知を受け取っていない」との主張があった場合に通知ログ要件を強化する

## 通知マトリクス

| トリガーイベント | 送信先 | チャネル | 必須内容 |
|---|---|---|---|
| `appointment.created` | 患者 | SMS + メール | 予約日時・クリニック名・キャンセルURL |
| `consent.agreed` | 患者 | メール | 同意日時・ポリシー全文 |
| `card.registered` | 患者 | SMS | カード末尾4桁・用途説明 |
| `authorization.succeeded` | 患者 | SMS | 予約確定・オーソリ金額・有効期限 |
| `authorization.failed` | 患者 | SMS | 予約未確定・カード再登録リンク |
| `authorization.expired` | 患者 + クリニック | SMS + メール | 失効理由・カード再登録リンク |
| `visit.confirmed` | 患者 | SMS | 来院確認済み |
| `capture.succeeded` | 患者 | メール | 請求金額・明細 |
| `cancellation.charged` | 患者 | SMS + メール | キャンセル料金額・計算根拠URL |
| `no_show` 検出 | クリニック | ダッシュボード通知 | 対象予約・キャンセル料回収ボタン |
| `dispute.opened` | クリニック + PF | メール | 紛争内容・対応期限 |

## 通知ログ保存要件（紛争証拠）

| フィールド | 必須 | 説明 |
|---|---|---|
| `notification_id` | ✓ | UUID |
| `appointment_id` | ✓ | 関連予約 |
| `event_type` | ✓ | トリガーイベント名 |
| `channel` | ✓ | SMS / email / push |
| `recipient` | ✓ | 送信先（マスク可） |
| `sent_at` | ✓ | 送信日時 |
| `delivered_at` | — | 配信確認（SMSは取得困難） |
| `content_hash` | ✓ | 送信内容のハッシュ（改ざん検知） |
| `status` | ✓ | sent / failed / retrying |

## 通知失敗時のフロー
1. 失敗を検知 → `status=retrying` でログ保存
2. 5分・15分・60分後に最大3回リトライ
3. 3回失敗 → `status=failed`、代替チャネルで送信試行
4. 全チャネル失敗 → OPERATIONS_OS.md の定期確認タスクで翌日確認

## 禁止事項
- `consent.agreed` の通知省略（同意証拠として送信必須）
- `authorization.failed` を「確認中」として非通知にする
- 通知失敗をサイレントに無視する（ログは必ず残す）

## 関連する状態
- EVENT_MODEL.md: 通知トリガーのイベント定義
- DISPUTE_RESPONSE.md: 通知ログを証拠として使用

## 関連する事故パターン
- `authorization.failed` を通知せず患者が予約確定と誤認して来院
- 通知ログに `content_hash` がなく「送った内容が違う」との主張に反論できない
- リトライ未実装で通知失敗を見逃し、患者が「連絡を受けていない」と主張
