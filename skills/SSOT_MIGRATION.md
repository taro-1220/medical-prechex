# SSOT_MIGRATION — SSOT移行ガイド

## 役割
dual writeや誤った正本参照を発見したときに、SSOTへ移行するための手順ガイド。
移行中の乖離リスクを最小化する。

## 管理対象
- SSOT違反パターンの識別方法
- 移行ステップのテンプレート
- 移行中の安全確保手順
- 移行完了の検証方法

## 書いてよい内容
- 違反パターンの分類と識別チェックリスト
- 安全な移行順序（読み取り移行 → 書き込み移行 → 旧パス削除）
- ロールバック条件と手順

## 書いてはいけない内容
- 実装コード
- 特定のORM・DBに依存した設定

## 更新ルール
- 新しい SSOT 違反パターンが発見されたときに追記する
- 移行完了後に「どこが難しかったか」を振り返りとして追記する

## SSOT違反パターン一覧

| パターン | 症状 | 識別方法 |
|---|---|---|
| dual write | 同じデータを2箇所に書いている | grep でフィールド名の更新箇所を探す |
| fallback延命 | 正本が取れない場合に古いストアで代替 | `??` / `|| cache.get()` のパターン |
| UI stateを正本 | localStorageやReact stateをDBの代わりに使う | `localStorage.getItem` をDB読み取りより前に参照 |
| Stripe状態無視 | Webhookを待たずにDBを更新 | APIレスポンスで直接 status を更新 |

## 移行ステップテンプレート

### Phase 1: 読み取りの移行（リスク低）
1. 新しい正本ストアから読み取るパスを追加（旧パスと並行稼働）
2. 両パスの結果を比較ログに出力（本番で乖離を観測）
3. 乖離が0になったら旧パスを削除

### Phase 2: 書き込みの移行（リスク中）
1. 新しい正本ストアへの書き込みを追加（旧ストアも継続して書く）
2. 新ストアへの書き込みが安定したことを確認
3. 旧ストアへの書き込みを削除

### Phase 3: 旧ストアの廃止（リスク低）
1. 旧ストアを参照している箇所がないことを確認
2. 旧テーブル・フィールドを deprecated マーク
3. 一定期間後に削除

## ロールバック条件
- 移行後に乖離件数が増加した場合
- 本番エラー率が移行前の2倍を超えた場合
- 患者からの問い合わせが急増した場合

## 関連する状態

## 関連する事故パターン
- 移行中に旧パスと新パスが競合し、データが上書きされた
- fallback の削除を忘れ、正本障害時に古いデータで動き続けた
