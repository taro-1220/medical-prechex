# TRANSACTIONAL_STATE_DESIGN — トランザクション状態設計ガイド

## 役割
複数ストア（DB + Stripe + ログ）にまたがる状態変化を安全に設計するためのガイド。
部分失敗・Webhook遅延・冪等性欠如による状態不整合を防ぐ。

## 管理対象
- 複数ストアにまたがる操作の設計パターン
- 冪等性キーの設計ルール
- 部分失敗時のリカバリ設計
- Webhook駆動の状態更新パターン

## 書いてよい内容
- 安全な操作順序（Stripe → Webhook → DB の原則）
- 冪等性キーの命名と生成ルール
- 部分失敗シナリオと対処方針
- アウトボックスパターンの適用ガイド

## 書いてはいけない内容
- 実装コード・ORMトランザクション構文
- 特定DBエンジンの設定

## 更新ルール
- 新しい複数ストア操作を追加する際に設計パターンを先にここで確認する
- 実際に部分失敗が発生した場合にパターンを追記する

## 基本原則

### Stripe → DB の順序
1. Stripe API 呼び出し（capture, cancel 等）
2. 成功を確認
3. DB 状態を更新
4. **禁止**: DB 更新後に Stripe API 呼び出し（DBは更新済みなのにStripeは未更新の状態が発生）

### Webhook 駆動の状態更新
1. DB には「操作開始」のみ記録（`capture_requested_at`）
2. Stripe API 呼び出し
3. Webhook 受信後に「操作完了」を記録（`captured_at`）
4. **禁止**: API レスポンスで `captured_at` を更新する（Webhook より先に更新すると二重になる）

### 冪等性キーの設計
- キー = `{operation_type}:{appointment_id}` （例: `capture:appt_xxx`）
- キーは操作ごとに1つ。再試行は同じキーで呼び出す
- キーの有効期間を明示する（Stripe は24時間）

## 部分失敗シナリオ

| シナリオ | DBの状態 | Stripeの状態 | リカバリ |
|---|---|---|---|
| Stripe API タイムアウト | 変更なし | 不明 | 同じ冪等性キーで再試行 |
| Stripe API 成功、DB更新前にサーバークラッシュ | 旧状態 | 新状態 | Webhook再送で復旧 |
| DB 更新成功、Webhook 未着 | 旧状態 | 新状態 | 定期的なStripe状態チェックで検出・修正 |
| capture 後の Webhook と API再試行が競合 | 重複更新の可能性 | 正常 | 冪等性キーでStripe側は防ぐ。DB更新は `updated_at` で排除 |

## アウトボックスパターンの適用
- 通知送信・外部API呼び出しは即時実行せず、outboxテーブルに記録してから非同期実行
- outboxレコードが残っている = 処理が完了していない
- 処理完了後にoutboxレコードを削除または `processed_at` を記録

## 関連する状態

## 関連する事故パターン
- Stripe API 呼び出し前に DB を更新し、API 失敗後に不整合が残った
- 冪等性キーがなく、タイムアウト後の再試行で二重 capture が発生
- Webhook 未着を検知できず、DB が stale のまま運用が続いた
