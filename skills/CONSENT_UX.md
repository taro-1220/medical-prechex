# CONSENT_UX — 同意取得UX設計ガイド

## 役割
キャンセルポリシー同意・カード登録目的説明など、法的効力が求められる同意UIの設計ガイド。
「同意した認識がない」「何に同意したか分からない」という患者の主張を防ぐ。

## 管理対象
- 同意画面の必須要素と禁止パターン
- 同意ログに保存すべき情報
- 同意フローの変更管理ルール
- アクセシビリティ要件（同意の有効性に影響）

## 書いてよい内容
- 同意画面ごとの必須要素リスト
- UX_RULES.md の同意UIルールの実装ガイド
- テキストの代替案評価基準（何が「曖昧な同意」か）
- 同意ログのペイロード設計ガイド

## 書いてはいけない内容
- 実装コード・UIコンポーネント定義
- 法律の解釈（→ MEDICAL_COMPLIANCE.md へ）
- キャンセル料計算（→ CANCELLATION_POLICY.md へ）

## 更新ルール
- 紛争で「同意UIの問題」が指摘された場合に更新する
- MEDICAL_COMPLIANCE.md に新しいコンプライアンス要件が追加されたときに更新する

## キャンセルポリシー同意画面の必須要素

### 表示必須
- [ ] ポリシーの全文（折りたたみ・省略禁止）
- [ ] ポリシーのバージョン番号と有効開始日
- [ ] キャンセル料が発生する具体的な条件と金額
- [ ] 無料キャンセル可能な期限

### ボタン・操作必須
- [ ] 同意ボタンのテキスト: 「上記内容に同意して予約を確定する」（曖昧な文言禁止）
- [ ] 同意しない場合の選択肢: 「予約をやめる」（押せないようにしない）
- [ ] 同意チェックボックスのデフォルト: 未チェック（強制）

### 禁止パターン
- デフォルトONのチェックボックス
- 「次へ」「確認」等、同意を明示しないボタン
- ポリシーを「詳細はこちら」で別画面に逃がす
- スクロールしないと同意ボタンが見えない設計（スクロール強制はOK）

## 同意ログのペイロード設計

同意イベント（`consent.agreed`）のペイロードに必須:

| フィールド | 説明 |
|---|---|
| `patient_id` | 同意した患者 |
| `appointment_id` | 対象予約 |
| `policy_id` + `policy_version` | 同意したポリシーのバージョン |
| `display_text_hash` | 表示した文面のハッシュ（改ざん検知） |
| `agreed_at` | 同意日時（ISO8601） |
| `ip_address` | 操作端末のIPアドレス |
| `user_agent` | 操作端末のブラウザ情報 |
| `consent_method` | `explicit_click`（明示的クリック必須） |

## カード登録目的説明の必須要素
- 「このカードはキャンセル料が発生した場合にのみ請求されます」
- 「正常来院の場合はカードへの請求はありません」
- 登録完了後: 「カードが登録されました（末尾XXXX）」の確認表示

## 同意フロー変更時のルール
- 同意ボタンのテキスト変更: MEDICAL_COMPLIANCE.md で確認後に変更
- ポリシー表示方法の変更: UX_RULES.md と整合性確認後に変更
- 同意ログのペイロード変更: EVENT_MODEL.md を先に更新

## 関連する状態
- EVENT_MODEL.md: `consent.presented` / `consent.agreed`
- CANCELLATION_POLICY.md: ポリシーバージョン管理
- DISPUTE_RESPONSE.md: 同意ログを証拠として使用

## 関連する事故パターン
- 「次へ」ボタンで同意を取得したため「同意した覚えがない」と主張された
- `display_text_hash` がなく、「表示した文面が異なる」との主張に反論できなかった
- IPアドレス未保存で「本人が操作した」を証明できなかった
