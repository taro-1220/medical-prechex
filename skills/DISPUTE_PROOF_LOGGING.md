# DISPUTE_PROOF_LOGGING — 紛争証拠ロギングガイド

## 役割
チャージバック発生時に証拠として使えるログを設計段階で組み込むためのガイド。
「ログがあれば勝てた」という事後の後悔を防ぐ。

## 証拠として有効なログの条件

| 条件 | 説明 |
|---|---|
| タイムスタンプ | サーバーサイドで付与（クライアント時刻は証拠として弱い） |
| 患者ID | 誰が操作したか特定できる |
| IPアドレス | 操作端末の特定 |
| User-Agent | 端末・ブラウザの特定 |
| 操作内容ハッシュ | 後から改ざんされていないことの証明 |
| 不変性 | ログの更新・削除禁止（append-only） |
| 保持期間 | 紛争提出期限 + マージン（最低3年） |

## ログの不変性を確保する設計パターン

### Append-only テーブル
- ログテーブルには `UPDATE` / `DELETE` を許可しない
- 訂正が必要な場合は「訂正ログ」を新しいレコードとして追記

### ハッシュチェーン（オプション）
- 各ログレコードに前のレコードのハッシュを含める
- 途中のレコードが改ざんされるとチェーンが壊れる

### 外部ストレージへの複製
- 重要ログ（同意・capture・キャンセル）はDB以外にも保存
- オブジェクトストレージ（S3等）に JSON で保存し、署名付きURLで取得可能にする

## 証拠パッケージ生成の設計

`dispute.opened` 受信時に以下を自動収集:

```
証拠パッケージ/
  consent_log.json        ← consent_logs テーブルから
  policy_text.txt         ← cancellation_policies.display_text (同意時バージョン)
  appointment_events.json ← event_logs where appointment_id = xxx
  notification_log.json   ← notification_logs where appointment_id = xxx
  stripe_events.json      ← Stripe API から PaymentIntent events を取得
```

各ファイルに `generated_at` と `data_source` を記録する。

## ログ保持期間の根拠

| ログ種類 | 保持期間 | 根拠 |
|---|---|---|
| 同意ログ | 10年 | 医療記録準拠 + 時効（消滅時効: 権利行使可能時から5年） |
| イベントログ | 7年 | 会計帳簿保存義務と同期 |
| 通知ログ | 5年 | 紛争時の参照期間 |
| 決済ログ | 10年 | Stripe の保持期間と整合 |

## 関連する事故パターン
- `agreed_at` はあるが `ip_address` がなく「本人操作」を証明できなかった
- ログを `UPDATE` で訂正したため改ざんを疑われた
- 証拠収集が手動作業で、期限までに間に合わなかった
